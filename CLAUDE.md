**API conventions (important for mobile client design):**
- Global prefix `api`, URI versioning → all routes are `api/v1/...`.
- Auth is **cookie-based for web, additionally bearer-token-capable for mobile**: JWT access + refresh tokens set as httpOnly cookies (web), `credentials: 'include'` on every fetch. `POST /auth/refresh` and `POST /auth/logout` also accept the refresh token via an optional `refreshToken` body field (`RefreshTokenDto`) now, so mobile clients (no cookie jar) can pass it explicitly; `/auth/refresh` returns the rotated `accessToken`/`refreshToken` in the JSON body for the same reason. `/auth/login` already returned tokens in the body pre-existing (unchanged). The JWT strategy's extractor already checked `Authorization: Bearer` as a fallback to the cookie, so protected routes needed no changes at all. Same `TokenService` rotation/jti logic for both transports.
- Global `ValidationPipe` (whitelist, forbidNonWhitelisted, transform).
- Global exception filter + response interceptor → consistent envelope (`{ success, data, message/error }` shape, mirrored in frontend's `handleResponse`).
- CORS locked to explicit origin allowlist (`CORS_ORIGINS` / `FRONTEND_URL` env, plus hardcoded prod domains).

**Seed users (`main.ts`, `seedInitialUsers`)**: existing pre-Claude mechanism — runs once at boot, only if the `users` collection is empty (`userCount === 0`), so it's a no-op after the first start. `INITIAL_USERS` is now env-aware (2026-07-22): `NODE_ENV=production` keeps the original real accounts (`supporttpt@gmail.com` admin / `tpttest@gmail.com` user) untouched; any other env (i.e. the dev Docker stack) seeds `admin@tpt.com` (role `admin`) and `user@tpt.com` (role `user`), both password `Password@123`. ## Mobile build-logic (convention plugins) — started 2026-07-22

After the first full clean build succeeded, started replacing the duplicated per-module `build.gradle.kts` boilerplate (identical `android { namespace; compileSdk 36; minSdk 24; Java 11 }` block, Compose setup, Hilt+KSP setup, and the 3-core-module-dependency baseline — copy-pasted near-verbatim across all ~15 modules) with a Now-in-Android-style `build-logic` included build + custom convention plugins. Confirmed this project was already feature-based-modularized as separate Gradle modules from day one (not a package-only monolith), so this is "de-duplicate boilerplate across an existing module graph," not a from-scratch modularization migration — lower risk. Sequenced deliberately behind the first successful build (agreed with Febin) so any breakage is isolated to build-logic itself, not conflated with AGP/Hilt/KSP/feature-code issues.

**Structure**: `mobile/build-logic/` is a separate Gradle included build (`pluginManagement { includeBuild("build-logic") }` in root `settings.gradle.kts`), with its own `settings.gradle.kts` (wires the SAME `gradle/libs.versions.toml` catalog into build-logic via `versionCatalogs { create("libs") { from(files("../gradle/libs.versions.toml")) } }` — one catalog, not two) and a single `:convention` module (`kotlin-dsl` plugin) registering 5 plugins under `build-logic/convention/src/main/kotlin/` (no package — top-level, per NiA convention):
- `thepowertrainer.android.application` (`AndroidApplicationConventionPlugin`) — `:app` only: common Android config + `targetSdk` (app-only concept).
- `thepowertrainer.android.library` (`AndroidLibraryConventionPlugin`) — bare library config (compileSdk/minSdk/compileOptions), the base every other plugin here builds on.
- `thepowertrainer.android.library.compose` (`AndroidLibraryComposeConventionPlugin`) — library + Compose, no Hilt. Used directly by `:core:designsystem`.
- `thepowertrainer.android.hilt` (`AndroidHiltConventionPlugin`) — Hilt + KSP + the `hilt-android`/`ksp(hilt-compiler)` dep pair. Kept standalone (not folded into the feature plugin) because `:core:network` needs Hilt too but isn't a feature module.
- `thepowertrainer.android.feature` (`AndroidFeatureConventionPlugin`) — the `:feature:*` template: library + Compose + Hilt + kotlin-serialization + the baseline dep set every feature module needs (`:core:common`/`:core:network`/`:core:designsystem`, navigation-compose, hilt-navigation-compose, lifecycle-runtime-ktx, activity-compose, coroutines, kotlinx-serialization-json, retrofit-core).

Shared config lives in plain (non-plugin) Kotlin files — `ProjectExtensions.kt` (`Project.libs` accessor exposing the version catalog to plugin code), `KotlinAndroid.kt` (`configureKotlinAndroid()` — the common `android {}` block), `AndroidCompose.kt` (`configureAndroidCompose()` — Compose setup) — matching Now in Android's pattern of composable helper functions rather than plugins applying other plugins directly.

**Critical constraint preserved**: none of the convention plugins apply `org.jetbrains.kotlin.android` — this project's AGP 9.3.0 has built-in Kotlin compilation, and that plugin is what caused the original kapt/AGP-9 build failures (see "Known gaps" #1 below). `configureKotlinAndroid()`'s kdoc calls this out explicitly so it isn't reintroduced by accident later.

**Pilot build-verified (2026-07-22) — hit and fixed 4 real AGP 9.3.0 API issues along the way** (all in `build-logic/convention`, none in app code):
1. `build-logic/convention`'s own JVM target had to be **17**, not 11 — AGP 9.3.0's `com.android.tools.build:gradle` artifact requires a Java-17-compatible consumer to even resolve as a `compileOnly` dependency. This is unrelated to and doesn't change the app modules' own Java 11 `compileOptions` (a totally separate setting, still 11 via `configureKotlinAndroid()`).
2. **`CommonExtension` is no longer generic in AGP 9.3.0** — dropped its 5-6 type parameters entirely. Fixed by writing `CommonExtension` with no type arguments (not `CommonExtension<*, *, *, *, *, *>`, the pattern older Now-in-Android-based examples use).
3. **The lambda-block DSL sugar (`defaultConfig { minSdk = 24 }`, `compileOptions { ... }`, `buildFeatures { compose = true }`) doesn't resolve on the plain `CommonExtension` type** in a compiled `.kt` file (as opposed to a `.gradle.kts` script) under AGP 9.3.0 — only plain property access does. Fixed by rewriting `configureKotlinAndroid()`/`configureAndroidCompose()` to use direct property-chain assignment (`defaultConfig.minSdk = 24`, `compileOptions.sourceCompatibility = ...`, `buildFeatures.compose = true`) instead of the block form.
4. **`com.android.build.gradle.LibraryExtension` (the class, from the `com.android.build.gradle` package) is the wrong type to `extensions.configure<>()` against under AGP 9's default `android.newDsl=true`** — it's deprecated and Gradle throws `Extension of type 'LibraryExtension' does not exist` at apply-time (a runtime failure, not a compile error) because the actually-registered extension is `LibraryExtensionImpl` implementing the newer `com.android.build.api.dsl.LibraryExtension` **interface**. Fixed by importing `LibraryExtension` from `com.android.build.api.dsl` (same package `ApplicationExtension` already correctly came from) in all three plugins that reference it (`AndroidLibraryConventionPlugin`, `AndroidLibraryComposeConventionPlugin`, `AndroidFeatureConventionPlugin`).

**Lesson for future AGP bumps**: AGP 9's "new DSL" (`android.newDsl=true`, default since 9.0) is a real, still-settling API surface — old Now-in-Android-pattern convention-plugin code (written against AGP 8.x's `CommonExtension<*,*,*,*,*,*>` and `com.android.build.gradle.LibraryExtension`) needs these adjustments, not a straight copy-paste. Always import Android DSL extension types from `com.android.build.api.dsl.*`, never `com.android.build.gradle.*`, going forward.

**Simplification flagged for verification**: the hand-written build files used AGP 9's newer `compileSdk { version = release(36) { minorApiLevel = 1 } }` block DSL; `configureKotlinAndroid()` uses the classic `compileSdk = 36` integer setter instead (guaranteed-stable API for programmatic `CommonExtension` configuration vs. guessing at the newer block DSL's exact Kotlin signature with no Maven Central access to verify it in this sandbox). Functionally equivalent for this app; revisit only if a specific minor SDK revision turns out to matter.

**New version-catalog entries** (`gradle/libs.versions.toml`): added `android-gradlePlugin`, `kotlin-gradlePlugin`, `compose-compiler-gradlePlugin`, `kotlin-serialization-gradlePlugin`, `ksp-gradlePlugin`, `hilt-gradlePlugin` — these are the actual plugin JAR coordinates (not IDs), needed as `compileOnly` classpath deps so `build-logic/convention`'s own Kotlin code can reference AGP/Kotlin/Hilt/KSP Gradle API types. Reuse the same `version.ref`s as the existing `[plugins]` entries — same caveat as the rest of this catalog (gap #1 below): not curl-verified against Maven Central in this sandbox, confirm on first build-logic sync.
